        participant U as User
        participant ST as ST (Epoch Vault)
        participant NAVO as NAV Oracle
        participant EPT as EPT

        Note over NAVO,ST: NAV updated every 5 min

## Frontend Integration

### Displaying User Positions

1. Query all ST Vault contracts for active + finalized epochs
2. For each vault: `balanceOf(user)` for shares, `sharePrice()` for current value
3. For each EPT: `balanceOf(user)` for EPT balance, `getUserCredits(user)` for accrued credits
4. For PointsTokens: `balanceOf(user)` for accumulated points across epochs

### Deposit Flow

```
1. User selects strategy + epoch
2. Frontend calls previewDeposit(usdcIn) → shows expected shares + EPT
3. User approves USDC spend on ST Vault
4. Frontend calls deposit(usdcIn, userAddress)
5. Listen for Deposit event → update UI
```

### Flash Loop Flow

```
1. User deposits USDC → receives ST + EPT
2. Frontend calls ArcX AMM swap: ST → USDC
3. Frontend re-deposits USDC proceeds → receives more ST + EPT
4. Repeat steps 2-3 (or batch atomically)
5. User keeps all EPT, sells all ST
```

### Redemption Flow

```
1. Check isFinalized() on ST Vault
2. For ST: call redeem(shares, userAddress)
3. For EPT: call previewClaim(userAddress) → show expected PointsTokens
4. Call claimPoints(userAddress)
5. Listen for PointsClaimed event → update UI
```

### Cross-Chain Deposit Flow

```
1. On source chain: approve USDC on SourceChainDepositEscrow
2. Call depositUSDC(amount, starknetReceiver, epochId)
3. Store depositId for tracking
4. Poll for DepositAcknowledged event (or timeout → show cancel option)
5. On Starknet: tokens appear in receiver's wallet
```

## Indexer / Subgraph Integration

### Key Events to Index

| Event | Source | Why |
|---|---|---|
| `Deposit` | ST Vault | Track TVL, user positions |
| `Redeem` | ST Vault | Track withdrawals |
| `NAVUpdated` | ST Vault | Build NAV history, compute returns |
| `EpochFinalized` | ST Vault | Trigger redemption UI |
| `CreditRateUpdated` | EPT | Build creditRate history |
| `CreditCheckpoint` | EPT | Track per-user credit accrual |
| `PointsFinalized` | EPT | Compute final points distribution |
| `PointsClaimed` | EPT | Track claims |
| `PointsTokenMinted` | PointsToken | Track supply growth |

### Derived Metrics

| Metric | How to compute |
|---|---|
| Strategy APY | `(finalNAV - initialNAV) / initialNAV` per epoch, annualized |
| User credit share | `getUserCredits(addr) / getTotalCredits()` |
| EPT implied cost | `1 - ST_amm_price / sharePrice()` |
| TVL per strategy | Sum of `currentNAV` across active epochs |
| Points earned per epoch | From `PointsFinalized` events |

## AMM Integration (ArcX AMM)

### Pool Setup

- **ST/USDC pool:** One per epoch per strategy. Uses a Pendle-style time-decay curve.
- **No EPT/USDC pool.** EPT is deposit-only and does not trade on any AMM.

### ArcX AMM Parameters

| Parameter | Description |
|---|---|
| `maturityTime` | Epoch end timestamp (curve converges ST toward NAV at this time) |
| `anchorRate` | Target exchange rate at maturity |
| `scalarRoot` | Controls curve steepness |
| `initialRate` | Starting exchange rate for the pool |

### Pool Credit Considerations

When EPT sits in a smart contract that holds it on behalf of users (e.g., an aggregator or custom vault), the contract address accrues credits. Post-finalization:

1. Admin calls `PointsCollector.claimPoolCredits(eptAddress, contractAddress)`
2. PointsCollector receives PointsTokens
3. Admin creates merkle distribution based on snapshots
4. Users claim via `PointsCollector.claimFromMerkle(proof, amount)`

**Snapshot timing:** Time-weighted snapshots taken during the epoch (multiple snapshots, not just one at epoch end). Exact timing is an operational parameter.

## Common Pitfalls

<AccordionGroup>
  <Accordion title="1. Stale NAV Deposits">
    Deposits revert if the NAV oracle hasn't updated in 30 minutes. If your integration experiences frequent reversions, check that deposits are timed after NAV updates (every 5 minutes).
  </Accordion>

  <Accordion title="2. Forgetting Credit Checkpoints">
    Every EPT transfer triggers a credit checkpoint. If you're building a custom contract that holds EPT, be aware that credits accrue to your contract address. You'll need a claim mechanism to distribute those credits to your users.
  </Accordion>

  <Accordion title="3. Epoch-Specific Tokens">
    ST and EPT are per-epoch. `ST-E007` and `ST-E008` are different contracts with different balances. Your frontend must track which epoch each token belongs to and display them separately. Use `getEpochInfo()` to identify tokens.
  </Accordion>

  <Accordion title="4. Cross-Chain Timing">
    Cross-chain deposits use arrival-time NAV, not deposit-time NAV. If NAV moves significantly during LayerZero transit (1--5 minutes typical, up to 30 minutes worst case), the user receives shares at a different price than expected. There is no slippage protection parameter currently.
  </Accordion>

  <Accordion title="5. Post-Finalization Only">
    `redeem()` on ST and `claimPoints()` on EPT both require `finalized == true`. These will revert during ACTIVE and MATURITY states. Check `isFinalized()` before calling.
  </Accordion>

  <Accordion title="6. Incremental Claims">
    `claimPoints()` can be called multiple times. The contract tracks `alreadyClaimed[addr]` and only mints the difference. This is safe for repeated calls but means you should not assume a single call captures all points. If global credit accounting updates between calls, additional points may become claimable.
  </Accordion>

  <Accordion title="7. PointsToken Minter Authorization">
    Each new EPT contract (each epoch) must be added as an authorized minter on the PointsToken contract. If `addMinter()` is not called for a new epoch's EPT, `claimPoints()` will revert when it tries to mint PointsTokens.
  </Accordion>

  <Accordion title="8. EPT Is Not Tradeable">
    EPT does not have an AMM pool and cannot be traded on any DEX. It is acquired only through deposits (and the flash loop). Do not attempt to create EPT liquidity pools.
  </Accordion>
</AccordionGroup>

## Contract Parameters Reference

| Parameter | Value | Set by | Contract |
|---|---|---|---|
| `MAX_NAV_STALENESS` | 30 minutes | Hardcoded | ST Vault |
| `depositFeeRate` | Configurable per strategy | Admin | ST Vault |
| `depositCutoff` | Configurable (default: 0) | Admin | ST Vault |
| `redeemFeeRate` | Configurable | Admin | EPT |
| `refundTimeout` | Configurable (expected ~30 min) | Admin | SourceChainDepositEscrow |
| `epochStart` | Set at deployment | Immutable | ST Vault |
| `epochEnd` | Set at deployment | Immutable | ST Vault |
| `epochId` | Set at deployment | Immutable | ST Vault, EPT |
| `strategyId` | Set at deployment | Immutable | ST Vault |
| `authorizedOracle` | Set at deployment | Admin | ST Vault, EPT |
| `maturityTime` | Set at pool creation | Immutable | ArcX AMM |
| `scalarRoot` | Set at pool creation | Admin | ArcX AMM |

## Deployment Checklist (Per Epoch)

Each new epoch requires deploying new contract instances:

<Steps>
  <Step title="Deploy ST Vault">
    Set `epochId`, `epochStart`, `epochEnd`, `strategyId`, `depositFeeRate`, `depositCutoff`.
  </Step>
  <Step title="Deploy EPT contracts">
    One per exchange. Set `epochId`, `pointsTokenAddress`, `redeemFeeRate`.
  </Step>
  <Step title="Link ST to EPT">
    Set `eptContracts` on ST Vault to point to new EPT instances.
  </Step>
  <Step title="Authorize minters">
    Call `addMinter(newEptAddress)` on each PointsToken contract.
  </Step>
  <Step title="Configure oracles">
    Set `authorizedOracle` on both ST Vault and EPT contracts.
  </Step>
  <Step title="Set up ArcX AMM pool">
    Create ST/USDC pool with Pendle-style time-decay parameters. Set `maturityTime` to `epochEnd`.
  </Step>
  <Step title="Seed initial liquidity">
    Provide initial ST/USDC liquidity to the AMM pool.
  </Step>
  <Step title="Verify">
    Call `getEpochInfo()` and `getCreditRate()` to confirm correct setup.
  </Step>
</Steps>
